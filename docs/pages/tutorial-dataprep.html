Worked Example

This work example demonstrates the step-by-step of creating the appropriate input files for the Small-Area Estimation (SAE) workflow. In the process, we will also address the various decisions that researchers need to make.

1 Preparation

1.1 Understanding the dataset

The survey dataset used is created based on the National Survey for Wales 2022-23. The National Survey for Wales is a cross-sectional survey, conducted annually to provide a snapshot of the Welsh population, covering topics include health, education, climate change, visits to the outdoors, public service satisfaction, material deprivation, internet access, and the Welsh language.

The survey dataset used is treated with stratified randomisation and randomised upsampling to both ensure data privacy and simluate a higher spatial resolution.

Note

In the original dataset, the Local Authority (LA) is the sole spatial identifier and the survey’s stratification variable. The example simulated dataset was created using:

  • Stratified randomisation: Survey responses are first split by Local Authority and their variables randomised column-wise to preserve variable totals by Local Authority.
  • Randomised upsampling: Survey responses are assigned a random MSOA that intersects with their recorded Local Authority to simulate a higher-resolution dataset for tutorial purposes.

Finally, population-level data aggregated at MSOA and LA levels come from the UK Census 2021 datasets, made available to download on the NOMIS UK website.

1.2 From Survey Indicators to Estimands

The first step in SAE is defining specific research questions that we hope to address using available survey data. For this tutorial with a transport analysis lens, we specifically want to estimate survey indicators related to commuting, bus usage, and active travel behaviour in Wales.

Then comes the formulation of the these survey indicators into estimands, depending on the target population, spatial (and temporal) scale of interest.

It could look something like this:

Outcome Target population Geographic scale
Mean commute distance among … Workers MSOA
Share of … dissatisfied with bus services Adults (aged 16+) MSOA
Share of … walking as main transport at least weekly All residents Local Authority

Specifying these elements early is critical as they dictate the technical requirements for the estimation process. For example:

  • Domain size: Estimating the mean commute distance among workers for all MSOAs generally requires known worker population totals in each MSOA to serve as the ‘denominator’ when calculating means. If not available, alternative estimation methods must be used to approximate the target group.
  • Weight: Estimating the share of adults dissatisfied with bus services generally requires knowing weights specifically calibrated for the adult population. Applying a general population weight would bias the results by including children. If the correct weights are not provided by the survey, the estimation approach must be adjusted to account for this.
  • Missingness: In a survey with approximately 12,000 respondents covering the whole of Wales, estimating the share of adults who walk at the LSOA level, totaling to nearly 2000 small areas, could leave the vast majority of LSOAs with zero or very few respondents. For SAE, this means the final estimates for these areas will be entirely model-based (inferred), which directly affects the precision and practical application of the data.
Tip

In practice, this step is reiterative, based on data availability and other restrictions. It is worth starting out with the most relevant estimands and refining them further after diagnosing the estimation results’ precision.

1.3 From Estimands to SAE Inputs

The next stage involves preparing the indicator variables and aligning them with the appropriate weights and domain sizes (i.e., target population totals in the small area scale of choice), as defined by the estimand.

Indicator Source variables Coding rule Target domain Domain size Weight
commute_distance TravWkDist numeric commute distance in miles, non-commuters are coded NA MSOA Worker population by MSOA SampleAdultWeight, analysis weight
poor_bus_satisfaction BusOverSat, Bus12M binary; 1 if fairly or very dissatisfied; 0 if not dissatisfied or did not use the bus in the last 12 months MSOA Adult population by MSOA WalesAdultWeight grossing weight
walk_weekly AtFrqWlk10 binary; 1 if walking for transport at least weekly Local Authority Total population by Local Authority WalesTravelWeight, grossing weight

In this dataset, weight variables are chosen based on the availability of auxiliary data:

  • SampleAdultWeight: analysis weight, used for sub-populations where an exact population total is unavailable (e.g., worker population by MSOA). This weight adjusts for sample design bias but does not scale the results to a specific population count.
  • WalesAdultWeight: grossing weight, used when the total adult population for a domain is known (e.g., adult population by MSOA). It scales the sample to match these real-world totals.
  • WalesTravelWeight: grossing-style weight, created by scaling SampleTravelWeight from the survey. This weight is used to scale the active-travel subsample (approx. 2,000 respondents) up to the total resident population in each Local Authority.

With this information we can proceed to process the survey dataset to produce inputs necessary for the SAE workflow.

2.3 Building Indicators

Define the missing-value rules used when constructing survey indicators, and calculate the indicators based on the rules set out above.

# Define a helper to handle the NSW specific missing codes
clean_nsw <- function(x) {
  ifelse(x %in% c(-99, -98, -9, -8, -7), NA, x)
}

survey_indicators <- nsw_raw |>
  transmute(
    unit_id,
    # Commute Distance
    commute_distance = as.numeric(clean_nsw(TravWkDist)),
    
    # Poor satisfaction (1 if dissatisfied, 0 if not or non-user)
    poor_bus_satisfaction = case_when(
      Bus12M == 2 ~ 0L,                         # Non-user
      clean_nsw(BusOverSat) %in% c(4, 5) ~ 1L,  # Dissatisfied
      is.na(clean_nsw(BusOverSat)) ~ NA_integer_,
      TRUE ~ 0L                                 # Satisfied
    ),
    
    # Weekly walking
    walk_weekly = if_else(clean_nsw(AtFrqWlk10) %in% 1:3, 1L, 0L)
  )

2.4 Selecting and building weight variables

Filter for the relevant weight variables from the survey.

survey_weights <- nsw_raw |>
  transmute(
    unit_id,
    SampleAdultWeight,
    WalesAdultWeight,
    SampleTravelWeight
  )

SampleTravelWeight is an analysis weight for the active-travel module. Because walk_weekly is defined over all residents at Local Authority level, we can create a grossing-style WalesTravelWeight by scaling SampleTravelWeight within each Local Authority so that the weights sum to the total resident population in each Local Authority.

survey_weights <- survey_weights |>
  # Attach LA IDs and Census population sizes
  left_join(select(survey_domains, unit_id, la_domain_id), by = "unit_id") |>
  left_join(select(domain_sizes_population_la, la_domain_id = domain_id, domain_size), by = "la_domain_id") |>
  
  # Calculate the grossing weight within each LA
  group_by(la_domain_id) |>
  mutate(
    WalesTravelWeight = SampleTravelWeight * (domain_size / sum(SampleTravelWeight, na.rm = TRUE))
  ) |>
  ungroup() |>
  
  select(unit_id, SampleAdultWeight, WalesAdultWeight, WalesTravelWeight)

2.5 Putting it together and missingness check

Join the coded indicators and calculated weights together for easy handling. Create a mapping object to link the indicator to its appropriate ‘companions’.

survey_inputs <- survey_domains |>
  left_join(survey_indicators, by = "unit_id") |>
  left_join(survey_weights, by = "unit_id")

indicator_map <- tibble::tribble(
  ~indicator,              ~target_domain, ~id_col,          ~weight,
  "commute_distance",      "MSOA",         "msoa_domain_id", "SampleAdultWeight",
  "poor_bus_satisfaction", "MSOA",         "msoa_domain_id", "WalesAdultWeight",
  "walk_weekly",           "LA",           "la_domain_id",   "WalesTravelWeight"
)

At this point, check for missingness overall (number of nulls per coded indicators) and at the chosen spatial scale (whether there are domains with no respondents).

# Calculate non-nulls and spatial coverage for each indicator at its own spatial scale
all_target_domains <- bind_rows(
  domain_sizes_adult_msoa |> mutate(target_domain = "MSOA"),
  domain_sizes_population_la |> mutate(target_domain = "LA")
) |> select(domain_id, domain_name, target_domain)

indicator_observations <- bind_rows(
  survey_inputs |>
    transmute(
      indicator = "commute_distance",
      domain_id = msoa_domain_id,
      value = commute_distance
    ),
  survey_inputs |>
    transmute(
      indicator = "poor_bus_satisfaction",
      domain_id = msoa_domain_id,
      value = poor_bus_satisfaction
    ),
  survey_inputs |>
    transmute(
      indicator = "walk_weekly",
      domain_id = la_domain_id,
      value = walk_weekly
    )
)

indicator_domain_coverage <- indicator_map |>
  select(indicator, target_domain) |>
  inner_join(all_target_domains, by = "target_domain") |>
  left_join(
    indicator_observations |>
      filter(!is.na(value)) |>
      count(indicator, domain_id, name = "non_missing"),
    by = c("indicator", "domain_id")
  ) |>
  mutate(
    non_missing = replace_na(non_missing, 0L),
    has_respondents = non_missing > 0
  )

# Summary
indicator_domain_summary <- indicator_domain_coverage |>
  group_by(indicator, target_domain) |>
  summarise(
    total_domains = n(),
    domains_with_no_respondents = sum(!has_respondents),
    total_non_missing = sum(non_missing),
    .groups = "drop"
  )

It is encouraging to see that the calculated indicators have strong spatial coverage overall, with only one MSOA without data. This means the subsequent SAE modelling can combine observed direct estimates with model-based adjustments, rather than relying almost entirely on modelled effects for many domains.

On the other hand, the number of non-missing observations reflects both survey routing and module design:

  • commute_distance has relatively few valid values because it is only asked of respondents who travel to work. Given that the target population is workers by MSOA, and no compatible worker domain-size file is available, this indicator should not be treated as a fully population-scaled estimate. This has implications for the choice of direct estimation method at the next step.
  • poor_bus_satisfaction is a routed variable asked only to bus users. However, it maintains near-full population coverage by coding non-users as 0 rather than null. Used together with the WalesAdultWeight variable, this approach is consistent with the estimand definition: the share of all adults dissatisfied with bus services.
  • walk_weekly comes from a subsampled active-travel module of about 2,000 respondents, which explains the low number of valid values. However, it has a companion WalesTravelWeight scaled to total population and a known domain size (total pop per LA) making it suitable for population-level estimation at LA scale.

2.6 Export

At this stage, the cleaned survey microdata and domain sizes files should look like this:

  • domain_sizes_adult_msoa.csv, for adult population by MSOA (from Census 2021);
  • domain_sizes_population_la.csv, for total resident population by LA (from Census 2021);
  • ind_weight_commute_distance.csv, with analysis adult weight;
  • ind_weight_poor_bus_satisfaction.csv, with grossing adult weight;
  • ind_weight_walk_weekly.csv, with active-travel module weight scaled to LA total population;
Tip

The next workbooks do not depend on local exports. However, if local exports are preferred, use the code block below to generate the input files into data/clean/ folder

output_dir <- "data/clean"
dir.create(output_dir, recursive = TRUE, showWarnings = FALSE)

# Create an input file for each indicator
for (i in seq_len(nrow(indicator_map))) {
  spec <- indicator_map[i, ]

  indicator_file <- survey_inputs |>
    transmute(
      unit_id,
      domain_id = .data[[spec$id_col]],
      indicator = .data[[spec$indicator]],
      weight = .data[[spec$weight]]
    )

  write_csv(
    indicator_file,
    file.path(output_dir, str_c("ind_weight_", spec$indicator, ".csv"))
  )
}

# Domain sizes used by the estimand-specific input files.
write_csv(domain_sizes_adult_msoa, file.path(output_dir, "domain_sizes_adult_msoa.csv"))
write_csv(domain_sizes_population_la, file.path(output_dir, "domain_sizes_population_la.csv"))
